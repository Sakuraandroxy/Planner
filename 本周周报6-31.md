# 本周周报（6/31 汇报版）

## 一、本周工作重点

本周主要围绕 **AirSim + Web Dashboard + MiMo VLM 闭环导航系统的演示效率优化** 展开，核心目标是减少汇报演示时的等待时间，让系统每一步的耗时来源变得清楚、可解释、可继续优化。

## 二、关闭 MiMo 思考模式

本周将 MiMo 调用格式改为官方推荐的 `max_completion_tokens`，并显式关闭思考模式：

```python
extra_body={"thinking": {"type": "disabled"}}
```

从终端日志看，关闭后每次 API 调用均显示：

- `explicit_thinking_param=disabled`
- `returned_reasoning=no`
- `reasoning_tokens=0`

说明当前慢的问题已经不是模型输出长思考链，而主要来自 **图像输入、API 推理本身、AirSim 取图和轨迹执行耗时**。

## 三、每步耗时统计

本次根据终端完整日志统计了前 5 步闭环耗时。日志显示 MiMo 思考模式已关闭，所有步骤 `reasoning_tokens=0`。

| Step | 总耗时 | AirSim取图 | 目标框API | 规划API | VLM总耗时 | 执行动作 |
|---|---:|---:|---:|---:|---:|---:|
| 1 | 43.63s | 8.20s | 11.47s | 9.62s | 23.52s | 11.55s |
| 2 | 30.60s | 8.03s | 3.56s | 7.29s | 13.29s | 8.83s |
| 3 | 34.62s | 12.88s | 2.84s | 7.45s | 11.76s | 8.87s |
| 4 | 31.40s | 8.32s | 2.67s | 6.67s | 11.13s | 11.52s |
| 5 | 33.01s | 14.00s | 2.87s | 8.15s | 13.25s | 5.33s |

平均耗时：

- **总耗时**：34.65s/步
- **VLM总耗时**：14.59s/步，占总耗时约 41.4%
- **目标框API**：4.68s/步；第 1 步为 11.47s，去掉第 1 步后约 2.98s/步
- **规划API**：7.84s/步，整体较稳定
- **AirSim取图**：10.29s/步，占总耗时约 30.2%，是当前最明显的非模型瓶颈
- **动作执行**：9.22s/步，占总耗时约 26.8%，主要受轨迹长度和移动速度影响

去掉第 1 步冷启动/异常偏慢影响后，第 2-5 步平均：

- **总耗时**：32.41s/步
- **VLM总耗时**：12.36s/步
- **目标框API + 规划API**：10.38s/步

## 四、核心结论

1. **关闭思考模式有效**：日志中 `reasoning_tokens=0`，说明没有继续消耗思考 token。
2. **当前每步仍慢，主要不是“思考模式”导致**：后续主要瓶颈变成 AirSim 取图、两次 VLM API 调用和动作执行。
3. **第 1 步目标框 API 仍有冷启动现象**：第 1 步 bbox API 为 11.47s，后续稳定在 2.67-3.56s，推测存在首次请求冷启动或服务端调度开销。
4. **规划 API 相对稳定**：每步约 6.67-9.62s，说明关闭思考后规划模型耗时已经可控。
5. **AirSim取图仍然偏慢**：每步约 8.03-14.00s，这部分不是 VLM 问题，后续应重点检查图像分辨率、AirSim RPC、DepthPerspective 获取方式和后台刷新。

## 五、本周代码调整

- 将 MiMo API 调用改为 `max_completion_tokens`。
- 增加 `extra_body={"thinking": {"type": "disabled"}}`，显式关闭思考模式。
- 增加每步耗时统计，拆分出目标框 API、规划 API、AirSim 取图和动作执行。
- 精简终端输出，只保留关键耗时：

```text
[TIMING Step 5/20] total=33.01s capture=14.00s bbox_api=2.87s planning_api=8.15s vlm_total=13.25s execute=5.33s thinking=disabled reasoning_tokens=0
```

- 目标检测阶段改为只要求 VLM 输出最终目标边界框，不再要求输出候选框数组，减少输出冗余。

## 六、后续计划

1. **继续压缩 VLM 调用次数**：当前每步仍有“目标框 API + 规划 API”两次调用，后续考虑目标跟踪或历史目标记忆，避免每帧重新找目标。
2. **重点排查 AirSim 取图慢的问题**：当前取图约 12s/步，已经接近或超过 VLM 单次规划耗时，需要检查高分辨率 RGB + Depth RPC 是否拖慢。
3. **演示前做 API 预热**：避免第 1 次 bbox API 出现 10s 以上冷启动等待。
4. **减少单步重新规划频率**：让 VLM 一次输出多步轨迹，由本地连续执行，只有目标丢失、接近终点或异常时再重新规划。
5. **加入目标记忆机制**：避免无人机到达目标附近后，因当前帧看不到原目标而误识别其他相似目标。

## 七、汇报总结

本周的核心进展是：**系统不再只是能跑，而是开始可测量、可解释、可优化**。通过关闭 MiMo 思考模式和加入耗时统计，已经明确当前闭环慢的主要来源：思考 token 已经清零，后续优化重点应转向减少 VLM 调用次数、优化 AirSim 图像获取和改进本地连续执行策略。
