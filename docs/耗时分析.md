# AirSim 闭环耗时分析

本文解释终端中 `[TIMING Step x/20]` 各字段的含义，以及为什么部分时间不是简单相加关系。

## 一、典型日志

```text
[TIMING Step 1/20] total=46.02s capture=7.20s bbox_api=2.92s planning_api=8.15s vlm_total=14.11s execute=24.39s thinking=disabled effort=low reasoning_tokens=0
```

这表示第 1 步闭环总耗时 46.02 秒，其中包括图像采集、目标框 API、规划 API、无人机执行等阶段。

## 二、字段含义

| 字段 | 含义 | 是否包含 API |
|---|---|---|
| `total` | 当前 step 从开始规划到执行结束的总耗时 | 包含 |
| `capture` | 从 AirSim 获取当前 RGB + DepthPerspective 深度矩阵的耗时 | 不包含 VLM API |
| `bbox_api` | 第一次 VLM 调用耗时：根据 RGB 和任务描述输出目标边界框 | 包含一次 API |
| `planning_api` | 第二次 VLM 调用耗时：结合 RGB、任务、目标深度输出候选轨迹和选择原因 | 包含一次 API |
| `vlm_total` | 整个 `planner.generate_candidates(...)` 的总耗时 | 包含两次 API |
| `execute` | AirSim 执行动作轨迹的耗时 | 不包含 VLM API |
| `thinking` | MiMo 的思考模式参数状态，例如 `disabled` 或 `not_set` | 仅配置状态 |
| `effort` | Step 模型的 `reasoning_effort`，例如 `low/high/default` | 仅配置状态 |
| `reasoning_tokens` | API 返回的 reasoning token 数量 | 统计信息 |

## 三、capture 包含什么

`capture` 对应 `run_airsim_web.py` 中这一段：

```python
capture_started = time.perf_counter()
frame, depth_meters = client.get_scene_and_depth_meters()
step_timing["capture_rpc"] = time.perf_counter() - capture_started
```

所以 `capture` 的边界是：

- 开始：当前 step 准备向 AirSim 请求最新观测。
- 结束：`get_scene_and_depth_meters()` 返回 RGB 图像和深度矩阵。

它主要包括：

- AirSim RPC 请求等待。
- 获取 RGB 图像。
- 获取 DepthPerspective 深度图。
- RGB 图像解码。
- 深度数据 reshape / 转 numpy。

它不包括：

- 目标框 API。
- 规划 API。
- 前端推送。
- 目标框区域深度计算。
- 无人机动作执行。

## 四、为什么 `vlm_total` 不等于 `bbox_api + planning_api`

`bbox_api` 和 `planning_api` 只统计两次远程 API 请求本身。

但 `vlm_total` 统计的是整个规划函数：

```python
vlm_started = time.perf_counter()
planner.generate_candidates(...)
step_timing["vlm_total"] = time.perf_counter() - vlm_started
```

因此：

```text
vlm_total = bbox_api
          + planning_api
          + 图片编码
          + bbox JSON 解析
          + bbox 坐标映射到深度图
          + 目标区域深度统计
          + 构造第二次规划 prompt
          + 规划 JSON 解析
          + 轨迹对象转换
          + 可能的重试/清洗
```

所以正常关系是：

```text
vlm_total >= bbox_api + planning_api
```

例如：

```text
bbox_api=2.92s
planning_api=8.15s
两者相加=11.07s
vlm_total=14.11s
差值约=3.04s
```

这 3 秒主要是本地图片编码、深度计算、prompt 构造和 JSON 解析，并不是额外的“思考时间”。

## 五、为什么 total 也不是各字段简单相加

当前日志只打印了核心字段，没有打印所有内部小阶段。

`total` 还可能包含：

- 深度图转前端预览图。
- RGB / 深度图推送到前端状态。
- 获取无人机位姿和碰撞状态。
- waypoint 转换。
- 记录历史上下文。
- 碰撞恢复。
- 少量 Python 调度和等待时间。

因此：

```text
total ≈ capture + vlm_total + execute + 其他小阶段
```

不是严格等于日志中几个字段的简单相加。

## 六、如何判断真正瓶颈

看日志时按优先级判断：

1. `execute` 大：飞行距离长或 `PLANNER_VELOCITY` 太低。
2. `bbox_api + planning_api` 大：模型/API 服务慢，或图像输入太大。
3. `capture` 大：AirSim 高分辨率 RGB + 深度图采集慢。
4. `vlm_total - bbox_api - planning_api` 大：本地图片编码、深度计算、解析或重试耗时偏高。

如果 `thinking=disabled/not_set` 且 `reasoning_tokens=0`，说明慢通常不是思考模式导致的。
