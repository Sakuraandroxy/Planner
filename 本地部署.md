# 本地 VLM 推理部署指南

## 架构概览

```text
笔记本（1660 Ti 6GB）
├─ AirSim 仿真
├─ run_airsim_web.py（规划器 + 控制闭环）
└─ 通过 LAN HTTP 发送 RGB/base64 + prompt
         ↓
  内网（有线 / Wi-Fi）
         ↓
服务器（4 × RTX 3080 Ti，12GB × 4）
├─ vLLM 推理服务（OpenAI-compatible API）
├─ Qwen3-VL-4B-Instruct
└─ 返回 bbox / planner JSON
```

笔记本只负责仿真和控制，模型推理在服务器完成。当前代码的 `VLMClient` 基于 OpenAI SDK，已原生兼容 vLLM API。

## 三、启动推理服务

### 方案 A：单卡模式（仅限显存充裕时）

单张 3080Ti 12GB 跑 `--max-model-len 8192` 大概率 OOM（KV cache 为负数），不推荐。仅供测试短上下文：

```bash
CUDA_VISIBLE_DEVICES=0 vllm serve /data/sakura/models/Qwen3-VL-4B-Instruct \
  --host 0.0.0.0 --port 8000 \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.95
```

### 方案 B：双卡张量并行模式（推荐 ★）

单卡 12GB 不够跑 8K 上下文 + KV cache，用张量并行将模型拆分到两张卡。

```bash
CUDA_VISIBLE_DEVICES=0,1 vllm serve /data/sakura/models/Qwen3-VL-4B-Instruct \
  --host 0.0.0.0 --port 8000 \
  --max-model-len 8192 \
  --tensor-parallel-size 2
```

**参数说明：**

| 参数 | 含义 |
|---|---|
| `CUDA_VISIBLE_DEVICES=0,1` | 指定使用第 0、1 号 GPU |
| `--host 0.0.0.0` | 允许局域网访问 |
| `--port 8000` | API 端口 |
| `--max-model-len 8192` | 最大上下文长度 8K（视觉 token 较多） |
| `--tensor-parallel-size 2` | 张量并行度 = 2（两张卡各放一半权重） |

**实测数据（3080Ti × 2）：**

| 指标 | 数值 |
|---|---|
| 单卡模型权重 | 4.48 GiB |
| CUDA Graph 显存 | 0.76 GiB |
| KV Cache 可用 | 3.83 GiB（约 55,792 tokens） |
| 最大并发（8K/请求） | 6.81× |
| 首次启动耗时 | ~54s（含 torch.compile 29s） |
| 二次启动耗时 | ~20s（torch.compile 缓存复用） |

> 首次启动后 torch.compile 缓存写入 `/home/sakura/.cache/vllm/torch_compile_cache/`，后续启动直接跳过编译阶段。

### 方案 C：分离服务模式（性能最优）

bbox 和 planner 分别部署在不同实例上，互不抢占。用卡 2、3 部署（卡 0、1 已被方案 B 占用）：

```bash
# 终端 1：bbox 检测（卡 2，轻量，短上下文够用）
CUDA_VISIBLE_DEVICES=2 vllm serve /data/sakura/models/Qwen3-VL-4B-Instruct \
  --host 0.0.0.0 --port 8001 \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.85 &

# 终端 2：planner 规划（卡 3，需更长上下文）
CUDA_VISIBLE_DEVICES=3 vllm serve /data/sakura/models/Qwen3-VL-4B-Instruct \
  --host 0.0.0.0 --port 8002 \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.85 &
```

分离模式需要在代码层面拆分 base_url，见第六节。

### 使用其他模型

四张卡有余量，可替换部署更强的模型：

```bash
# 例如 Qwen2.5-VL-7B-Instruct
CUDA_VISIBLE_DEVICES=2 vllm serve /data/sakura/models/Qwen2.5-VL-7B-Instruct \
  --host 0.0.0.0 --port 8003 \
  --max-model-len 8192 --gpu-memory-utilization 0.90
```

## 六、分离 bbox / planner API 的代码改动（方案 C 专用）

当前 `VLMClient` 和 `PlannerConfig` 只有一套 `base_url` / `model_name`。分离后可分别指定 bbox 和 planner 的服务地址。

### 6.1 PlannerConfig 增加字段

编辑 `planner/config.py`，在 `PlannerConfig` 中新增：

```python
# bbox 检测 API（未设置则回退到 base_url / model_name）
bbox_base_url: str = ""
bbox_model_name: str = ""
```

### 6.2 Planner 增加独立的 bbox VLMClient

编辑 `agent/planner.py`，在 `__init__` 中：

```python
self.vlm_bbox = VLMClient(self._bbox_config())

def _bbox_config(self):
    from planner.config import PlannerConfig as Cfg
    c = Cfg()
    c.base_url = getattr(self.config, "bbox_base_url", "") or self.config.base_url
    c.model_name = getattr(self.config, "bbox_model_name", "") or self.config.model_name
    c.api_key = self.config.api_key
    c.temperature = self.config.temperature
    return c
```

然后将 `estimate_target_depth()` 中的 `self.vlm.call(...)` 替换为 `self.vlm_bbox.call(...)`。

## 八、常见问题

**Q：服务器启动报 CUDA OOM**

单卡 12GB 无法跑 `--max-model-len 8192`。改用方案 B（双卡张量并行）或方案 A（降低上下文长度至 4096）。

**Q：笔记本 curl 连不上**

- 检查服务器防火墙是否放行端口
- 确认笔记本和服务器在同一子网（内网）
- 服务器 IP 是否变化

**Q：模型推理结果与云端 API 不一致**

- 检查 `temperature` 设置
- 确认 prompt 一致
- `Qwen3-VL-4B-Instruct` 能力弱于云端大模型，复杂场景需调 prompt

**Q：图传太慢**

1080p PNG base64 约 2-3MB，内网带宽（100Mbps+）传输 < 0.3s。如需更小，可在 `capture_scene_depth()` 中使用 JPEG 编码：

```python
# 替换 PNG 为 JPEG（体积缩小 5-10 倍）
rgb_buffer = BytesIO()
frame.save(rgb_buffer, format="JPEG", quality=90)
```
