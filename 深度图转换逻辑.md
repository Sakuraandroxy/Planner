# 深度图转换逻辑

## 一、当前方案：目标框 + 原始深度矩阵

现在系统不再让 MLLM 从像素颜色里反推精确深度，而是采用更稳定的两阶段方法：

1. MLLM 只负责在 RGB 图中定位任务目标，输出归一化目标框 `bbox_norm=[x1,y1,x2,y2]`；如果有多个同类目标，输出 `candidates` 列表。
2. 程序用 AirSim 返回的原始 `DepthPerspective` float32 深度矩阵，在每个 bbox 区域内计算真实米制深度。
3. 如果任务包含“较远/远处/最远/far”等词，程序从候选中选择 `target_depth_median` 最大的目标。
4. 将最终目标的中位深度、最近深度、平均深度写入后续规划 prompt。

这样做的好处是：视觉模型擅长找目标，程序擅长从深度矩阵里精确取数，避免让 MLLM 读热力图文字或解码颜色导致误差。

## 二、数据来源

AirSim 的 `DepthPerspective` 返回 float32 二维数组：

```text
深度矩阵[y, x] = 该像素对应位置到相机的距离，单位：米
```

代码入口：

- `sim/airsim_client.py#get_depth_meters()`：返回原始米制深度矩阵，给 bbox 深度估计使用。
- `sim/airsim_client.py#get_depth_image()`：返回 8-bit 灰度图，只给前端展示使用。
- `sim/airsim_client.py#get_labeled_depth_heatmap()`：返回带文字标注的彩色深度热力图，作为 MLLM 辅助视觉参考。
- `sim/airsim_client.py#get_depth_image_vlm()`：兼容旧名称，内部转调 `get_labeled_depth_heatmap()`。

## 三、完整数据流

```text
AirSim
  │
  ├─ RGB Scene 图
  │   └─ 发给 MLLM：定位任务目标 bbox / candidates
  │
  ├─ DepthPerspective float32 深度矩阵
  │   └─ get_depth_meters()
  │       └─ 在每个 bbox 内取有效深度像素
  │           ├─ target_depth_median：目标中位深度
  │           ├─ target_depth_min：目标最近深度
  │           └─ target_depth_mean：目标平均深度
  │
  └─ get_labeled_depth_heatmap()
      └─ 发给 MLLM：规划阶段的辅助深度图

Planner
  │
  ├─ estimate_target_depth()
  │   ├─ 调 MLLM 获取 bbox/candidates
  │   ├─ parse_target_candidates()
  │   └─ estimate_depth_in_bbox()
  │   └─ choose_target_estimate()
  │
  └─ generate_candidates()
      └─ 将目标 bbox 深度 + 场景深度统计写入 prompt
```

## 四、目标框深度估计

新增模块：`agent/target_depth.py`

核心步骤：

```python
bbox = [x1, y1, x2, y2]
region = depth_meters[y1:y2, x1:x2]
valid = region[(region > 0.1) & (region < 1000.0) & np.isfinite(region)]
target_depth_median = np.median(valid)
target_depth_min = np.min(valid)
target_depth_mean = np.mean(valid)
```

默认使用 `target_depth_median` 作为目标距离的主要参考，因为它比最小值更抗噪声、比平均值更不容易被背景深度拉偏。

为避免 bbox 覆盖道路、天空或背景，当前实现会优先使用 bbox 中央 50% 区域的中位深度，并写入 `target_depth_center_median`。如果 RGB 图和深度矩阵分辨率不同，程序会先把 RGB 坐标缩放到深度矩阵坐标，再裁剪深度区域。

## 五、深度偏小的常见原因

如果实际目标约 50m，但估计只有 20~30m，优先检查下面几项：

1. **bbox 坐标系错了**：部分 MLLM 会输出 0~1000 坐标，而不是原图像素坐标。当前 prompt 已改为要求 `bbox_norm` 归一化坐标，解析器也兼容 0~1000 坐标。
2. **RGB 与深度图尺寸不同**：RGB bbox 不能直接裁深度矩阵，必须按宽高比例缩放。当前 `scale_bbox_to_depth()` 会打印并使用 `depth_bbox`。
3. **bbox 框到近处物体或道路**：整框中位数可能混入前景道路。当前优先使用 bbox 中心区域中位数。
4. **“较远汽车”被框成近车**：如果画面有多辆车，日志里要看 `candidates` 是否包含远车；任务含“较远/远处/最远/far”时程序会选深度最大的候选。

运行时重点查看：

```text
[TARGET DEPTH] image_size=(RGB宽, RGB高) depth_shape=(深度高, 深度宽)
[TARGET DEPTH] candidates: target=... raw_bbox=... rgb_bbox=... depth_bbox=... median=... min=... mean=...
[TARGET DEPTH] chosen: target=... rgb_bbox=... depth_bbox=... median=...
```

## 六、为什么不用 R+G 编码当主方案

旧方案曾尝试把深度写入 RGB 像素：

```text
(R × 256 + G) / 100 = 实际米数
```

这个方法理论上能表达厘米级深度，但实际依赖 MLLM 准确读取像素颜色并做计算。多数 API 会对图片重采样、压缩或内部归一化，视觉模型也不擅长逐像素数值解码，所以结果不稳定。

当前方案把数值计算移回 Python，只让 MLLM 做目标定位，因此更可靠。

## 七、规划 prompt 中的深度信息

`agent/prompt_builder.py` 会把深度上下文格式化为文本，例如：

```text
深度提示：汽车bbox=[120,80,260,210]，置信度=0.86，目标中位深度=8.54m，目标最近深度=7.92m，目标平均深度=8.80m，画面中心最近深度=6.30m，画面中心平均深度=9.10m。
```

规划时应优先遵循：

1. `目标中位深度`：目标主体距离。
2. `目标最近深度`：碰撞安全下限参考。
3. `画面中心深度`：目标不可见或 bbox 不可靠时的备用参考。
4. 第二张深度热力图：辅助判断空间结构，不作为精确数值主来源。

## 八、失败与回退

如果目标不可见、MLLM bbox JSON 解析失败，或 bbox 内没有有效深度像素：

- 系统不会中断主规划。
- 日志会输出 `[TARGET DEPTH] skipped: ...`。
- prompt 回退到场景/中心区域深度统计和深度热力图。

## 九、方法命名约定

- `get_depth_meters()`：真实米制深度矩阵，给程序计算使用。
- `get_labeled_depth_heatmap()`：带文字标注的彩色深度图，给 MLLM/人类参考使用。
- `estimate_target_depth()`：先让 MLLM 找 bbox，再用深度矩阵计算目标深度。
- `get_depth_image_vlm()`：旧接口兼容别名，不建议新代码继续使用。
