### 2. 任务解析功能（TaskParser）

本周新增 **TaskParser 模块**：任务开始前先调用一次纯文本 VLM API（不传图），将自然语言任务拆成结构化阶段。

**多阶段示例：**
用户输入：飞到前方三叉路口旁，左转，直行直到找到红色汽车，靠近它

TaskParser 输出（4 阶段）：

  stage 1: 直走到三叉路口旁

  stage 2: 左转

  stage 3: 找到第一辆车

  stage 4: 靠近第一辆车

> **优势：** VLM 更擅长理解中文复合指令、省略表达和阶段关系；支持动态扩展，无需修改代码即可适配新任务句式。

> **缺点：** 额外花费1-2s时间解析

### 3. 上下文管理（ContextManager）

本周实现上下文模块两个功能：**目标记忆**和**状态机**

**目标记忆：**
- 首次发现任务目标时记录目标类别、bbox、深度、朝向和世界坐标。
- 后续帧根据历史目标信息判断当前检测是否仍是同一个目标，减少追错同类目标的问题。

**状态机：**

| 状态 | 触发条件 | 作用 | 为什么必要 |
|---|---|---|---|
| INIT_DETECT | 首次可靠检测到任务目标 | 建立目标记忆，记录类别、bbox、深度和世界坐标 | 没有初始锁定，后续无法判断“是不是同一个目标” |
| TRACKING | 当前帧仍能看到锁定目标 | 正常进入 planner API，根据当前目标继续规划 | 表示闭环处于稳定跟踪阶段，可以继续执行任务 |
| LOST_OR_OCCLUDED | 当前帧看不到目标，或检测结果不像原目标 | 暂停普通目标跟踪，避免把新同类目标当成原目标 | 目标短暂遮挡或出画时，如果继续追当前帧结果，容易追错目标 |
| REDETECT | 还没有形成有效目标记忆 | 继续调用检测逻辑寻找任务目标 | 任务刚开始或检测失败时，需要和“目标丢失”区分开 |

这四个状态对应闭环中最基本的四种情况：**刚锁定、稳定跟踪、目标丢失、尚未锁定**

### 4. 四向环视重定位（Relocalizer）

当目标丢失系统进入 **Relocalizer 流程**。

**作用：** 防止 VLM 在看不到目标时盲目输出 forward 动作；同时根据历史深度/世界坐标判断丢失目标的可能方向。

**工作流程：**
1. 无人机原地旋转四次每次90°，依次看向 4 个方向。
2. 每个方向抓帧，调用 VLM 检测目标。
3. 综合 4 个方向的检测置信度和深度，选择最可能是原目标的方向。
4. 输出恢复方向（yaw）和候选数量，引导无人机重新对准目标。

**代价：** 单次重定位约 60s（4 个方向 x 调 VLM + 旋转）

## 二、OnFly 调研

*OnFly: Onboard Zero-Shot Aerial Vision-Language Navigation toward Safety and Efficiency*(IROS'26)是面向无人机的机载零样本 AVLN 框架，重点不是让 VLM 直接规划轨迹，而是把 VLM 约束在目标定位上，再由本地几何规划器生成可飞轨迹。

### 1. 与当前系统的核心区别

| 对比项 | 当前系统 | OnFly |
|---|---|---|
| VLM 职责 | 两阶段：先输出目标边界框，再输出候选轨迹 | 只输出目标在图像中的 2D 位置 |
| 轨迹生成 | VLM planner 给出多条候选轨迹/动作 | 程序根据目标位置和深度地图计算一条轨迹 |
| 输出形式 | `bbox + candidates/actions` | `(px, py)` 目标像素坐标 |
| 规划主体 | VLM 参与轨迹规划 | 本地 Fast Planner 负责轨迹规划 |
| 候选数量 | 多候选，可比较不同路径 | 单轨迹，规划器直接给出当前最优解 |
| 安全约束 | 主要依赖 prompt、深度和后处理 | ESDF 地图、障碍膨胀、重规划窗口共同约束 |

**核心差别：** 我们当前让 VLM 同时承担“看见目标”和“规划怎么飞”两个任务，因此输出更灵活，但也更容易出现不稳定动作、格式错误或绕路问题。OnFly 则把 VLM 的作用压缩为“告诉程序目标在哪里”，后续轨迹完全由确定性的本地规划器完成，所以速度和稳定性更好，但灵活性更弱。

### 2. OnFly 的关键模块

- **Decision Agent**：高频调用，负责从当前图像中定位目标，只输出目标像素坐标，不输出动作。
- **Monitoring Agent**：低频调用，约每 2 秒判断一次当前任务状态，例如继续、停止或目标丢失。
- **Hybrid Memory**：混合记忆机制，保留首帧、历史关键帧和最新帧，让 VLM 有任务上下文，同时保持 prompt 前缀稳定，提高 KV-cache 命中率。(我们现在是基于目标的类别、坐标等纯文本)
- **语义-几何验证**：把 VLM 输出的 2D 目标点结合深度图转换为 3D 目标点，并用局部语义特征和障碍物膨胀做安全检查。
- **本地规划器**：基于 ESDF 地图和 receding-horizon 方式生成一条碰撞规避轨迹，不依赖 VLM 直接输出动作。

### 3. OnFly 单步推理速度

| 模块 | 单步耗时 |
|---|---:|
| Decision Agent | 0.81s/step |
| Monitoring Agent | 1.15s/step |

速度快的主要原因不是模型更强，而是系统架构更合理：VLM 只做目标定位，两个 Agent 分频运行，视觉特征可共享，Hybrid Memory 提高 KV-cache 利用率，最终轨迹由本地规划器快速生成。

### 4. 对我们系统的启发

- 后续可以弱化 VLM planner，让 VLM 主要做目标检测、语义定位和任务状态判断。
- 轨迹规划应逐步转向本地几何规划器，避免让 VLM 直接生成 `forward/left/right` 动作。
- 保留我们当前“多候选轨迹”的优势，用本地规划器生成多条可行轨迹，再通过规则或打分器选择。
- OnFly 的缺点也很明确：规划器偏硬编码，每步只输出一条轨迹，灵活性不如我们当前的多候选方案。
